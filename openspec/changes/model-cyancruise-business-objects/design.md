# 设计：CyanCruise 金蝶业务对象连续建模

## 总体方案

本变更先不引入代码实现，而是把金蝶业务对象建模口径落成可审查文档。对象按 CyanCruise 主循环推进：先补齐运行审计类对象“Agent执行记录”的布局，再建立用户画像后最关键的“职业测评记录”。

## Agent执行记录布局

“Agent执行记录”是单据对象，面向管理员、排障人员和后续运营统计。结构化字段用于筛选、排序和成本统计；输入输出上下文用大文本承载，避免过早拆分高变动 AI payload。

表单分为四块：

1. 基本信息：单据编号、用户 ID、智能体、场景、触发方式、状态。
2. 执行时间与消耗：开始时间、结束时间、耗时、token 和成本。
3. 执行步骤：分录展示步骤状态和说明。
4. 输入输出：折叠或页签展示输入、输出和错误信息。

## 职业测评记录对象

“职业测评记录”使用单据对象而不是基础资料，因为每次提交都是一次可追溯业务事件。单据头保存量表、状态、摘要、主要结果和完成时间；答案明细分录保存题目、选项、维度、分值和有效性；完整答案和结果继续保留 JSON 快照。

该对象与既有测评能力的边界保持一致：测评记录可以写入用户职业画像的测评块，但不得覆盖画像中的目标岗位、简历、面试或职业计划块。

## 简历记录对象

“简历记录”使用单据对象承接 `/cc001/resume/*` 和 `ResumeRecordDto` 的业务语义。单据头保存简历 ID、用户、标题、目标岗位、文件标识、版本、状态、诊断分数和时间戳；解析正文和完整诊断结果用大文本字段承载；关键词和诊断建议使用分录展示，便于在金蝶列表和表单中查看关键问题。

文件引用只保存稳定 `file_key`，不得把临时预览 URL、签名参数或访问凭据写入业务对象。简历记录写入用户职业画像时，只更新画像的简历块，不覆盖测评、面试、职业计划或新用户引导数据。

## 数据安全

Agent 执行记录的输入内容不得保存 access token、Authorization header、签名 URL 或客户私有凭据。职业测评记录只保存用户提交的测评答案和结果，不保存无关设备信息或前端运行上下文。简历记录只保存稳定文件标识、解析正文和诊断结果，不保存临时文件访问凭据。

## 后续扩展

完成简历记录后，建议继续建立“今日任务”，再进入“职业计划”“模拟面试记录”和“求职助手会话”。这个顺序与 CyanCruise 主循环和现有迁移优先级一致。
