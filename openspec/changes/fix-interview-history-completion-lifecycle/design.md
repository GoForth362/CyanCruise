## Context

当前面试会话在开始时即写入 `InterviewStorage` 并标记为 `ONGOING`，这是后续逐题回答接口按 `interviewId` 读取上下文的必要条件。但 `listByUser` 和 `listPage` 直接返回全部会话，前端又把进行中记录渲染成“继续面试”，因此临时会话错误地成为长期历史。

前端历史操作同时存在“查看结果”和“查看问答”。“查看结果”调用 `openInterview(..., "result")`，在报告尚未加载或状态同步异常时会进入答题渲染分支，形成截图中的“正在准备问题”。问答入口已经能够读取同一记录的持久化消息，所以两个入口没有必要并存。

## Goals / Non-Goals

**Goals:**

- 历史分页只包含已完成且至少有一条用户回答的有效面试。
- 历史记录只保留“查看问答”和“删除记录”，不允许恢复进行中的会话。
- 主动结束有回答会话时完成并生成复盘；零回答会话直接删除。
- 离开面试路由时自动完成有回答会话或删除零回答会话。
- 新面试开始前清理当前用户遗留的 `ONGOING` 会话。

**Non-Goals:**

- 不取消面试进行期间的服务端临时会话存储，因为逐题接口需要稳定的 `interviewId`、所有权和消息上下文。
- 不修改面试业务对象字段或新增数据表。
- 不保证浏览器被操作系统强制终止、断电或完全断网时退出请求一定到达服务器；通过下一次开始时清理遗留会话兜底。

## Decisions

1. 历史准入条件集中在 `InterviewApplicationService`。
   - `listByUser` 与 `listPage` 均只返回 `status=COMPLETED` 且消息中至少存在一条 `USER` 回答的会话。
   - 理由：不能只依赖前端过滤，否则 API、管理入口或其它客户端仍会读到进行中记录。

2. `/cc001/interview/finish` 调整为丢弃无效会话。
   - 前端只在零回答退出时调用该入口；后端删除会话和关联消息并返回 `OK`。
   - 有回答的主动结束仍使用 `/guided/finish`，保证状态、复盘和分数完整。

3. 开始新会话前删除同一用户全部 `ONGOING` 会话。
   - 理由：浏览器异常关闭时退出请求可能无法送达；用户下次开始练习时清理可以阻止半截会话永久积累。
   - 已完成历史不受影响。

4. 路由离开时执行统一后台收尾。
   - 有回答调用 `/guided/finish`；零回答调用 `/finish` 丢弃。
   - 立即失效本地操作令牌并释放语音、摄像头和计时资源，异步响应不再恢复旧页面。
   - `pagehide` 使用 `fetch(..., keepalive: true)` 发送同一请求，作为浏览器关闭时的尽力而为处理。

5. 历史记录移除 `open/result` 操作。
   - 文字和全景记录均只保留问答详情与删除操作。
   - 问答详情可展示已保存问答；当前面试结束后的复盘仍在结束页面即时展示。

## Risks / Trade-offs

- [历史过滤需要读取消息判断有效回答] → 每页查询前会检查候选会话消息；当前数据量和固定 10 条分页可接受，并通过测试锁定行为。
- [离开页面时复盘生成较慢] → 路由切换不等待结果；服务端收到请求后继续完成，历史页只会在完成后显示该记录。
- [异常关闭请求未送达] → 下一次开始面试前清理同一用户遗留的 `ONGOING` 会话。
- [旧的零回答 `COMPLETED` 数据仍在数据库] → 历史准入要求至少一条用户回答，因此旧空记录立即不可见；用户可通过既有治理方式后续清理物理数据。
- [修改当前未提交的面试代码] → 仅在面试生命周期、历史动作和对应测试位置做局部补丁，保留正在开发的智能体复盘逻辑。
