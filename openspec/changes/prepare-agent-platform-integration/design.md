## Context

当前金蝶 Agent 开发平台中已存在用户画像、信息检索、考研规划、就业规划和简历优化等基础智能体，并已建立升学规划、就业岗位、简历面试三个知识库。目标架构扩展为通用应用助手、生涯发展主调度、用户画像、人格分析、信息检索、就业规划、简历优化、模拟面试、考研规划、保研规划和留学规划共十一类智能体能力。

当前阶段先完成就业规划任务流蓝图及 CyanCruise 服务端接入。任务流只接收服务端汇总的当前用户画像、深度画像、目标岗位和所选简历证据，返回结构化就业计划；用户在应用中主动点击生成后，服务端才会保存到既有职业计划存储。平台凭据和用户身份不得暴露到浏览器。

## Goals / Non-Goals

**Goals:**

- 让平台用户可通过一个主入口完成职业发展咨询、考研、保研、留学、就业、简历优化和模拟面试。
- 让每个任务流只使用职责相符的知识库和智能体，并让输出可被主流程汇总。
- 建立固定的会话变量和输出结构，避免节点之间依赖不明确的自然语言内容。
- 为后续由 CyanCruise 后端调用 Agent 平台预留稳定的请求、会话、身份和结果保存边界。

**Non-Goals:**

- 不由 Agent 平台直接读取或写入 CyanCruise 数据库。
- 不新增职业计划或今日行动数据库表，不改变既有存储后端选择。
- 不允许浏览器直接调用 Agent SDK、持有智能体凭据或决定最终用户身份。
- 不把政策、院校分数线、薪资或招聘信息当作没有来源和更新时间的固定事实。

## Decisions

### 决策：采用“一个主调度流程 + 多个专业任务流”

生涯发展主调度任务流作为平台的统一职业发展入口，负责识别问题、准备共享上下文、调用专业任务流并汇总答案。考研、保研、留学、就业、简历优化、就业面试、升学面试和职业测评分析各自保留独立任务流，既可被主流程调用，也可单独试运行。

不让五个智能体相互自由对话。自由协作难以追踪输入、无法稳定复现，也不利于后续 API 集成。

### 决策：人格分析、用户画像和信息检索作为流程准备节点

人格分析节点根据职业测评结果输出兴趣、能力、工作和风险偏好；用户画像节点汇总基础背景、经历、目标、人格结果、缺失字段和已识别意图；信息检索节点只从授权知识库检索依据，输出带来源类型的资料摘要和“资料不足”标识。三者不直接给出最终职业结论。

这样可以避免同一问题被多个规划节点重复猜测用户背景或编造事实。

### 决策：主流程保留并行专业分析，但按意图控制分支

当用户询问“考研还是就业”时，同时运行考研规划和就业规划；当用户询问保研、留学、简历或面试时，只运行对应专业任务流。单一场景问题只运行对应分析，减少无关输出。

若平台当前版本暂不支持条件分支，先保留三个并行分析节点，但在每个节点提示词中规定：与本节点无关时只输出“本次不适用”，主汇总节点不得把该内容当作建议。

### 决策：统一会话变量和结果结构

主流程使用 `question` 作为原始问题，使用 `profileSummary`、`intent`、`missingInfo`、`retrievalContent`、`studyResult`、`employmentResult`、`resumeResult` 和 `finalAnswer` 作为节点输入输出。变量名可按平台控件实际限制调整，但含义不得改变。

专业节点均返回“结论、依据、风险、下一步行动、资料不足”五部分。主汇总节点只基于各节点输出和检索依据生成面向学生的中文答案。

### 决策：后续后端接入使用服务端适配层

正式集成时，CyanCruise 页面调用自有 `/cc001/*` 业务接口；后端适配层保存平台会话标识、传递经过服务端确认的用户身份、调用 Agent 平台主任务流，并将确认后的结构化计划保存到职业计划和今日行动边界。浏览器不得直接保存或使用 Agent 平台凭据。

平台侧 H5 发布可作为当前阶段演示入口，但不等同于已完成应用集成。

### 决策：就业规划通过官方 Agent SDK 调用已发布智能体

服务端使用 `SDKClient.create().getAgentService().runAgent(...)` 调用配置项 `cc001.agent.platform.employment.agentNumber` 指向的已发布智能体。任务流编码仍保留在通用请求契约中，供后续 HTTP 发布形态或链路追踪使用；SDK 调用时以智能体编码作为实际入口。配置只来自运行时系统属性或环境变量，不写入前端与源码。

### 决策：任务流输出直接对应现有职业计划 DTO

就业规划任务流 SHALL 返回 camelCase JSON，至少包含 `targetRole`、`startStateSummary`、`phases`、`weeklyPlan` 和 `dailySuggestions`。服务端严格校验这些字段后才保存为 `CareerPlanRecordDto`；失败时保留已有计划并返回普通中文可重试提示。完整节点和提示词见 `employment-planning-taskflow.md`。

### 决策：重新生成路线图采用阶段级执行锁与二次确认

首次生成路线图不需要风险确认。已有路线图存在未开始阶段时，页面必须先展示红色风险确认窗口，说明执行中或已完成阶段及其进度会被保留，只有未开始阶段会被智能体结果替换；用户确认后才调用智能体。前端只使用当前路线图中实际存在的阶段任务判断局部执行状态，忽略不属于当前阶段的历史勾选记录。服务端按阶段状态合并新旧路线图，并保留正在执行阶段对应的本周计划、每日建议和路线图版本；只有所有阶段都已开始或完成时才拒绝生成。

### 决策：长规划摘要在页面端按语义分点展示

`startStateSummary` 继续保留为服务端结构化计划中的完整文本，页面端按分号、换行或完整句拆分，并根据“阶段、基础、缺口、目标偏好、待确认、时间风险”等关键词添加普通中文主题标签。页面标题下只保留简短说明，完整摘要放入独立概览卡，避免长文本挤占首屏且不改变智能体原始数据。

路径规划页不再额外展示仅包含目标岗位和固定一年周期的“路线概览”卡片。目标与范围已经由当前情况、页面说明和阶段路线图表达，删除重复卡片可以让每日计划、本周计划、总进度和阶段内容保持连续。

### 决策：每日任务复用已有今日任务对象并与路线图来源动作绑定

每日计划复用 `v620_cc_career_task`，不新增数据库表。任务编码包含路线图版本和日期，`task_key` 保存来源动作编码，`parent_task_id` 保存阶段编码，`due_date` 保存计划日期，`status` 保存完成状态，`description` 保存面向页面的任务内容。服务端生成当天任务前先读取同一计划版本的历史任务：未完成项优先顺延，已完成来源动作直接排除，再从当前阶段动作、短周期动作和阶段达成条件中按顺序补足。完成接口同时更新任务状态，并在阶段全部来源动作完成后推进职业计划阶段状态。

页面为当前路线图维护一个阶段游标。无论用户从阶段卡片还是每日计划完成最后一个来源动作，页面都根据真实阶段任务完成数重新计算首个未完成阶段：已完成阶段保持只读完成状态，下一阶段立即成为当前聚焦并显示“进行中”。每日任务完成接口返回后，页面重新读取服务端路线图摘要，使本地游标、服务端阶段状态和下一次每日任务生成来源保持一致；当天已经生成的任务批次仍按日期稳定保存，不因阶段切换被随机替换。

## Risks / Trade-offs

- [知识库资料过少或过期] → 为每份资料记录来源与更新时间；检索节点在无依据时输出“当前资料不足”。
- [所有分支都并行执行导致答案冗余] → 优先配置意图条件；条件不可用时采用“不适用”输出约束。
- [节点变量不一致导致主汇总为空] → 先在每个任务流单独试运行，固定变量名并在主流程配置中逐项映射。
- [平台侧智能体与任务流职责重复] → 智能体负责提示词、知识库授权和专业能力；任务流负责编排、变量传递和输出。
- [未来接入时泄露平台凭据或越权读取数据] → 仅允许后端适配层调用平台 API；身份由 Cosmic 服务端上下文确认，不信任浏览器传入的用户标识。

## Migration Plan

1. 在 Agent 开发平台补齐三个专业任务流并分别试运行。
2. 校准现有职业发展规划任务流的节点变量、知识库授权、并行分支和主汇总提示词。
3. 使用固定测试问题完成平台侧验收并发布 H5 或平台入口。
4. 平台侧稳定后，再创建后端接入 change，实现服务端适配层、身份校验、会话关联、结构化结果保存和应用页面联调。
5. 若平台任务流不可用，禁用其发布入口或回退到已有 CyanCruise 聊天能力；不影响现有应用业务接口。

## Open Questions

- 当前租户的任务流是否支持按意图条件分支、子任务流调用和智能体节点调用；若不支持，需要以多个独立任务流加主入口提示词替代。
- 平台发布后提供的是 H5 链接、内嵌入口、API 调用，还是多种方式；后端接入前需要确认正式 API 文档和鉴权方式。
- 用户画像节点在平台阶段是否只做会话内摘要，还是允许写入平台会话变量；未接后端前不得视为 CyanCruise 的正式用户数据。
