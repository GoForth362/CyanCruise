## Context

`index.html` 当前在脚本执行前直接输出旧 CareerLoop 入口头部、身份面板和状态卡片。浏览器完成 HTML/CSS 首次绘制后，`app.js` 才动态加载 `app-runtime.js`；运行脚本随后在默认模式中通过 `renderChromeMode()` 隐藏这些区域，并在身份解析完成后渲染 URL 指定的功能页。这个时序使旧入口壳在网络或脚本解析稍慢时短暂可见。

现有运行脚本仍通过固定 DOM id 使用身份调试输入框和状态字段，因此直接删除整个 DOM 结构会造成空引用，也会破坏显式调试模式。

## Goals / Non-Goals

**Goals:**

- 默认用户模式从首次可绘制时就隐藏调试入口壳，消除历史页面闪现。
- 删除“求职主循环”“主循环状态”等历史用户可见文案。
- 保留现有功能路由、身份解析、页面挂载、WebAPI 请求和显式调试模式。
- 用静态验证锁定首屏隐藏和历史文案清理结果。

**Non-Goals:**

- 不重构页面路由、身份服务或 `app-runtime.js` 的整体架构。
- 不删除调试模式所需的 DOM id 和输入能力。
- 不改变后端接口、数据模型、业务规则或依赖。

## Decisions

1. 调试入口头部、路由导航和状态区在静态 HTML 中直接带上 `chrome-hidden`。
   - 理由：样式表位于 `<head>`，浏览器首次绘制前即可应用隐藏规则，不依赖运行脚本下载和执行。
   - 备选方案是仅在 `init()` 中隐藏；该方案正是当前闪现窗口的来源，因此不采用。

2. 保留调试 DOM 结构及现有 id，只替换历史标题与说明。
   - 理由：`app-runtime.js` 仍会更新身份、岗位、准备度、简历和面试状态，并为开发身份输入框绑定事件。保留结构能最大限度降低功能回归风险。
   - 备选方案是删除结构并把所有访问改成空值安全；改动面更大，且会取消现有调试能力，因此不采用。

3. `renderChromeMode()` 继续作为显式调试模式的显示开关。
   - 默认模式保持隐藏；`?ccDebug=1` 时移除 `chrome-hidden` 并恢复调试导航与状态信息。

4. 工作台中仍有业务价值的状态面板改名为“求职准备概况”，不删除其数据内容。
   - 理由：其数据属于当前功能，不是旧页面壳；只清理历史命名即可。

5. 扩展现有 `validate-routes.js`，检查默认隐藏类和禁止的历史文案。
   - 理由：无需引入新依赖，并能在现有静态资源验证命令中防止回归。

## Risks / Trade-offs

- [调试区域被永久隐藏] → 保留 `renderChromeMode()` 的移除隐藏类逻辑，并验证 `ccDebug=1` 路径使用的 DOM 结构不变。
- [修改当前未提交的运行脚本时覆盖用户工作] → 只对精确文案行做局部补丁，不改相邻业务逻辑。
- [浏览器继续使用旧缓存] → 更新相关静态资源版本号，并在交付时说明需重新部署静态资源。
- [仅靠人工确认闪现消失] → 在静态验证脚本中检查首屏 HTML 默认隐藏状态，作为确定性门禁。

## Migration Plan

1. 更新静态入口的默认隐藏类和调试文案。
2. 更新运行时业务标题与静态验证。
3. 更新缓存版本并执行 JavaScript、路由、OpenSpec 和聚焦构建验证。
4. 部署更新后的静态资源；如需回滚，恢复本 change 涉及的静态文件版本即可，后端无需回滚。
