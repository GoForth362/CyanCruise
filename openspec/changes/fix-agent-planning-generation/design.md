# 设计：修复智能规划生成链路

## Context

金蝶智能体 SDK 既支持按智能体编码执行，也支持按任务流编码直接执行。当前升学配置在加载时主动清空任务流编码，使后端只能进入对话智能体。运行日志显示，对话智能体返回了 `Thought`、`Action` 和 `Action_input`，但没有产生任务流最终输出。现有解析器随后可能选中用户输入回显，最终因缺少路线字段而失败。

求职规划与升学规划共享相似的智能体调用和路线结构，同样存在平台包装层、Markdown、字符串化 JSON 或工具调用过程被误识别的问题。

## Goals / Non-Goals

### Goals

- 运行时配置了任务流编码时，后端直接执行任务流。
- 只把满足路线契约的最终对象交给业务校验。
- 兼容常见 `data`、`result`、`answer`、`output` 包装及字符串化 JSON。
- 不清空用户已保存路线，不用规则内容冒充智能体结果。

### Non-Goals

- 不在代码中硬编码租户专属任务流编码或访问凭证。
- 不替代金蝶平台中的智能体、任务流发布和授权配置。
- 不修改规划页面的视觉设计。

## Decisions

### 1. 任务流编码优先

升学配置不再清空 `taskFlowCode`。客户端已有“任务流编码存在则直接执行，否则执行智能体”的能力，因此保留该配置即可复用现有调用路径。智能体编码继续作为没有任务流编码时的兼容回退。

### 2. 共享解析器递归解包

共享解析器从完整响应中枚举 JSON 候选对象，递归检查常见包装字段。只有包含 `targetRole`、`phases`、`weeklyPlan` 等路线特征的对象才参与映射。`Action_input`、用户输入回显和仅含 `mode` 的请求对象不会被当成规划结果。

### 3. 业务校验保持严格

解析器只负责识别与解包，升学和求职服务继续执行各自的完整性校验。考研的未来十二个月、多阶段和真实性约束不降低。

### 4. 配置缺失时明确失败

仓库不保存租户任务流编码。部署环境必须配置对应 `taskFlowCode`。当既没有可用任务流结果，也没有合格智能体结果时，返回普通中文错误并保留旧数据。

## Risks / Trade-offs

- 外部任务流输出字段若完全不符合现有路线契约，仍会被拒绝；这是防止虚假数据持久化的必要边界。
- 对响应进行候选扫描会增加少量解析成本，但相较智能模型调用耗时可忽略。
- 运行环境需要管理员填写并发布正确的任务流编码，否则仍会回退到对话智能体。

## Verification

- 单元测试覆盖直接对象、`data/result/answer/output` 包装、字符串化 JSON、多个相邻 JSON、工具调用输入回显。
- 服务测试验证任务流编码未被清空、错误时旧路线不被覆盖。
- 使用 JDK 8 和仓库内 `gradlew.bat` 构建。
