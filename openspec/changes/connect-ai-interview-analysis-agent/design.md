## Context

当前 `InterviewApplicationService` 已让文字面试和全景面试共用 `InterviewStorage`：问题与回答保存在 `InterviewMessageDto`，结束时由 `InterviewAiService` 生成 `InterviewReportDto`。问题在于 `InterviewAiService.report()` 吞掉所有异常，解析失败后调用 `InterviewAiHelper.fallbackReport()`，按回答数量计算固定分数并生成固定文案。该结果随后写入会话、最终分数和用户画像，页面无法区分真实 AI 与规则占位结果。

简历诊断已经形成可复用模式：场景级 Agent 配置、苍穹 Agent SDK/服务端任务流客户端、严格 JSON 验证、失败时抛出用户可理解的不可用错误，并且只展示和保存真实 Agent 结果。面试应采用相同边界，但使用独立配置和面试报告契约。

## Goals / Non-Goals

**Goals:**

- 让文字面试与全景面试根据实际问题和回答生成同一结构的真实 AI 报告。
- 报告包含总分、五项分数、每项评分依据、带问答证据的优点和改进建议。
- 复用 Agent 平台 SDK/任务流客户端，不在浏览器暴露地址、令牌或智能体编号。
- AI 不可用或输出无效时不伪造、不保存分数，并允许用户稍后对同一记录重试生成。
- 保持 JDK 1.8、现有 WebAPI 路径和面试存储兼容。

**Non-Goals:**

- 不上传或分析摄像头视频、面部表情、视线、肢体动作和原始音频。
- 不迁移 IPD 的 Spring Boot、Vue、uni-app 实现。
- 不在代码中写入生产令牌、智能体编号或本地绝对路径。
- 不在本次 change 中搭建 Agent 平台上的知识库和智能体；平台侧资产继续由 `prepare-agent-platform-integration` 管理。

## Decisions

### 1. 使用场景级 `InterviewAnalysisAgent`，而不是继续在应用服务直接调用通用 `AiGateway`

新增面试分析接口及 Agent 平台实现，配置前缀使用 `cc001.agent.platform.interview`。默认工厂选择方式与简历诊断一致：优先苍穹 Agent SDK，必要时使用服务端任务流客户端；未配置时返回明确不可用。这样可让面试智能体独立发布、授权和迭代，也避免复用简历智能体编号。

备选方案是只删除 `fallbackReport()` 并保留通用 `AiGateway`。该方案改动较小，但无法满足“一系列智能体接入”的场景隔离，也不能复用已验证的任务流配置与 SDK 调用方式，因此不采用。

### 2. 由后端组装可信上下文并要求证据化 JSON

请求上下文由服务端从已鉴权会话和消息存储构造，包括目标岗位、难度、面试方式、有效回答数量和按顺序排列的问题/回答。模型只负责分析，不接收或决定 `userId`、`interviewId` 与数据归属。提示要求优点和改进项引用候选人的实际回答片段或明确指出哪一题缺少证据，不得编造经历。

### 3. 扩展现有报告 DTO，保持存储向后兼容

在 `InterviewRadarScoreDto` 或报告中增加五项评分依据集合，优点和改进继续使用 `InterviewAdviceItemDto(title, detail)`。JSON 持久化存储可兼容新增字段；读取旧报告时允许依据为空，但新生成报告必须完整。不开新表、不引入依赖。

### 4. 结构校验失败等同 AI 失败，不做规则评分

校验至少覆盖：总分和五项分数均为 0～100 的整数；有效回答数与服务端统计一致；总结非空；优点和改进项均非空且标题、说明完整；五项评分依据非空。客户端最多进行有限重试，仍失败则抛出“AI 面试分析暂时不可用，请稍后重试”。会话可以结束，但 `report` 和 `finalScore` 保持空值，避免错误数据进入历史与画像。

### 5. 两个结果页复用完整报告组件

全景面试仍只上传语音转写后的文字回答；完成后调用同一个 `/cc001/interview/finish-report`。普通和全景结果页都展示总分、五项得分与依据、做得好的地方和改进方向。请求失败时保留当前会话，展示可重试提示，不渲染 0 分或默认文案。

## Risks / Trade-offs

- [智能体任务流尚未发布或授权] → 使用独立配置开关和明确错误；部署检查中验证智能体编号、授权与任务流输入输出。
- [模型引用不存在的经历] → 提示词要求证据，后端校验结构；用户界面明确分析来源为本次文字问答。语义真实性无法只靠结构校验完全证明，需在智能体提示和验收样例中持续测试。
- [旧报告缺少评分依据] → 读取旧记录保持兼容，页面只在字段存在时展示；新报告严格要求完整依据。
- [结束会话后报告生成失败] → 保留已保存问答和已结束状态，用户可从历史记录再次请求报告，不丢失练习材料。
- [上下文过长] → 服务端按字符上限截取并优先保留完整的最近问答，且不记录完整上下文到错误日志。

## Migration Plan

1. 发布并授权面试分析智能体/任务流，约定 `question` 输入和结构化 JSON 输出。
2. 部署后端代码并配置 `cc001.agent.platform.interview.*`；先在测试环境验证 SDK 调用。
3. 部署带新结果展示和失败重试状态的 webapp 静态资源。
4. 旧的已保存基础复盘不自动重写；后续如需清理，使用单独的数据治理 change 按来源证据处理。
5. 回滚时恢复旧代码即可，但不得重新启用虚假评分；可临时将功能置为明确不可用。

## Open Questions

- 生产环境面试智能体的 `agentNumber`/任务流编码由平台管理员在部署时提供，不进入仓库。
- 是否对历史中已由 `fallbackReport()` 生成的报告做批量识别和清理，需要先确认现网数据规模，留给后续数据治理变更。
