## Context

统一升学陪伴智能体已接收 13 个任务的真实表单请求，但任务流未被约束为按页面实际可提供的资料生成结果；其 `NEED_MORE_INFO` 也可能没有具体说明。前端多数表单未在调用前校验核心字段，考研择校还遗漏了请求 DTO 已支持的本科专业。

## Goals / Non-Goals

**Goals:**

- 让每个页面只在其核心资料缺失时阻止调用，并明确指出字段。
- 将本科专业纳入考研择校请求；为 13 个任务提供一致、可发布的智能体提示词和 JSON 信封规则。
- 保持智能体为唯一生成来源；服务端继续校验 taskType、状态和 DTO。

**Non-Goals:**

- 不在后端生成虚构的规则、模板或兜底结果。
- 不改变 WebAPI 路径、用户隔离或已保存的分析记录。
- 不在应用内替用户配置或发布金蝶平台的智能体。

## Decisions

1. 在前端建立按接口路径识别的最小字段校验表。它与页面文案一致，调用前直接显示中文缺项，避免把可预防的空请求交给智能体。替代方案是由每个 ApplicationService 各自校验；不采用，因为 13 个任务的页面反馈仍会滞后且容易分叉。
2. 考研择校新增“本科专业”并提交 `major`。该字段已存在于请求 DTO，补齐它不会破坏现有接口。
3. 平台提示词采用“任务类型 + 最小输入 + 结果结构”的单一契约文档。所有任务在核心输入齐全时必须 `OK`；非核心资料只能写进结果的 `missingInfo`，不得返回 `NEED_MORE_INFO`。替代方案是分别维护 13 份提示词；不采用，避免任务信封和状态规则不一致。
4. 服务端对空白 `NEED_MORE_INFO` 增加可诊断提示，明确这是智能体未按任务契约给出缺项，而不是让用户猜测。

## Risks / Trade-offs

- [平台提示词尚未发布] → 代码会准确提示并阻止空请求，但无法代替平台输出；部署清单要求重新发布智能体。
- [字段规则随页面演进失配] → 将字段表、OpenSpec 和提示词同改，并以页面路由校验测试覆盖。
- [智能体返回 Markdown 或多段 JSON] → 保持服务端现有严格解析，提示词明确只返回单个 JSON 对象。

## Migration Plan

1. 部署前端和后端 JAR，重启实际承载服务的 JVM。
2. 在平台更新并发布统一升学陪伴智能体的系统提示词。
3. 使用每个页面的核心输入各生成一次，核验 13 个 taskType 的 `OK` 结果。
4. 若需回滚，恢复原前端资源和 JAR；用户已保存的草稿与成功记录不受影响。

## Open Questions

- 无。现有请求 DTO 和结果 DTO 是平台提示词的唯一字段来源。
