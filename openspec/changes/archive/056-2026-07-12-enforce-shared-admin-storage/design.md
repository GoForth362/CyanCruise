## Context

共享使用模式中，`10.0.0.8` 上的 `DebugApplication` 是唯一的 CyanCruise 服务节点，其他电脑只通过浏览器访问该节点。当前存储工厂在 PostgreSQL 后端未生效时会为部分能力返回进程内存实现；管理台前端又将读取接口失败降级为空列表。这会让不同服务实例或异常请求看起来像是不同管理员拥有不同数据。

现有 Cosmic 登录上下文可以识别平台用户和管理员角色，现有 PostgreSQL 存储已覆盖用户画像、通知与管理员治理表。本次同时将订阅额度从进程内存迁移至 `cc_subscription_quota`，以 `(user_id, template_id)` 作为唯一键。改造必须兼容 JDK 8、Cosmic/KDDT 模板和现有 KAPI 路由，不引入新依赖。该变更不直接迁移 IPD 实现：IPD 来源为“无新增迁移来源”；目标模块为调试启动模块、CyanCruise 业务应用和 webapp；数据映射为现有用户标识、管理员治理记录、订阅额度和业务状态继续映射到 PostgreSQL 的同一 schema。

## Goals / Non-Goals

**Goals:**

- 让共享服务实例以完整 PostgreSQL 配置作为业务状态、订阅额度和管理员治理状态的唯一持久化数据源。
- 让普通用户请求始终以已解析的平台用户标识作为数据归属边界，管理员请求始终以已解析的平台管理员身份作为授权边界。
- 让任意管理员从同一服务和同一数据库读取一致的管理数据，管理写入可被其他管理员和相关用户读取。
- 让管理后台明确展示加载失败，不把失败当作空数据。
- 让调试启动在局域网模式下明确区分监听地址、对外访问地址和数据库配置，并在配置不足时尽早失败。

**Non-Goals:**

- 不支持多台 `DebugApplication` 自动复制内存数据或做多主集群。
- 不改变金蝶平台管理员角色的来源与授权模型。
- 不为每个开发者自动创建数据库、schema 或迁移既有内存数据。
- 不迁移 IPD 的 Spring Boot、JPA、Flyway、Vue、uni-app 或 JWT 实现。

## Decisions

### 共享模式禁止业务与治理状态回退到内存

共享模式由显式运行时开关标识，并要求 `cc001.storage.backend=postgresql` 及 URL、用户名、密码、schema 完整。存储工厂在共享模式下创建任何已迁移的业务或治理存储前都验证完整性；缺失或无法连接时返回稳定配置失败，不创建内存替代实现。

选择该方案是因为内存回退会将部署错误伪装为正常运行，且无法跨服务实例共享。备选方案是保留内存实现并添加告警；该方案仍允许请求命中不同节点后读取不同结果，无法满足管理员一致性。

本地单机测试可保留显式开发内存模式，但它必须与共享模式互斥，并在页面或日志中清楚标识，不得作为共享访问地址的后端。

### 单一对外节点与显式公开地址

`DebugApplication` 从本地配置或环境变量读取 Web 监听地址、公开访问地址和 PostgreSQL 配置，并在启动前写入运行时系统属性。局域网共享运行使用 `server.bind.ip=0.0.0.0` 或主机局域网地址，`domain.contextUrl` 使用主机可达 URL；客户端不运行调试服务。

选择显式公开地址而不是由请求头推断，是为了避免 Cosmic 登录完成后将客户端重定向到 `127.0.0.1`。备选方案是由反向代理改写请求头；当前调试部署没有稳定的反向代理边界，因此不采用。

### 以已解析身份决定读取与写入范围

WebAPI 身份边界继续解析当前 Cosmic 登录上下文。普通用户业务请求忽略或校验外部传入的用户标识，并以解析到的 `userId` 查询、写入。管理员拥有普通用户入口能力，同时只能在解析到管理员等价角色时访问 `/cc001/admin/*`；管理员列表和治理操作不按管理员账号分片，而是针对统一 PostgreSQL 状态。

选择服务端身份边界而不是浏览器 localStorage，是因为浏览器可被切换、清理或伪造，且无法在跨设备时保持安全一致。备选方案是以请求体中的 `adminId` 过滤数据；这会导致管理员间视图不一致，也与现有身份规格冲突。

生产页面初始化时始终调用 `/cc001/identity/current` 获取当前服务端身份，即使浏览器窗口、Cookie、sessionStorage 或 localStorage 中已有用户标识，也不得据此跳过服务端解析。身份调用失败或返回无效状态时页面按“需要身份”处理，不继续使用缓存用户调用受保护接口。

首页草稿、展开状态和路线进度等浏览器状态使用服务端确认的 `userId` 作为命名空间。同一浏览器检测到账号变化时，旧版未分区缓存迁移到原用户命名空间，当前用户只读取自己的命名空间，避免把上一账号的未提交状态展示给新账号。

### 管理端以“失败”区别于“无数据”

管理前端服务模块将 `whoami`、用户列表及其他读取请求的异常转换为结构化失败状态，页面显示普通中文的“管理数据暂时无法加载”及可重试操作。只有服务端成功返回空分页时，才显示“暂无用户数据”。

选择该方案是为了缩短配置和路由故障的定位时间。备选方案是维持 `safe(..., emptyPage)` 兜底；它会隐藏服务端错误并误导管理员。

### 用户搜索与管理员身份标注

用户管理搜索由前端输入事件触发服务端分页查询，服务端统一匹配昵称、用户 ID、学校和专业。用户身份类型不写入 `cc_admin_user`，列表返回前通过金蝶平台权限服务实时补充管理员标记，避免账号权限调整后数据库副本过期，也避免把普通用户登记表误用为授权表。

### 数据与部署迁移

持久化目标保持为 PostgreSQL `cyancruise` 数据库及配置指定 schema，管理员治理数据使用既有 `cc_admin_*` 表，订阅额度使用 `cc_subscription_quota` 表及用户和模板组合主键。首次共享部署前由非破坏性 DDL 创建表和索引；运行时默认不执行破坏性 DDL。已经存在于内存实例中的记录不具备可靠的跨进程导出通道，需按测试数据重新创建，不承诺自动迁移。

## Risks / Trade-offs

- [共享 PostgreSQL 不可用会阻断业务和管理端] → 共享模式 fail fast 并展示明确中文错误；部署前验证数据库连接与表结构。
- [多个调试实例仍注册到同一 Cosmic 环境] → 约定仅主机运行共享实例；本地开发使用独立访问地址和独立数据库/schema，避免作为共享 KAPI 节点。
- [旧浏览器缓存保留 API 模式或会话] → 在错误提示中提供刷新/重新登录建议，并使用版本化静态资源。
- [管理员账号在平台中的角色解析失败] → 保持 fail closed，返回无权限而不是授予管理能力；通过管理员身份接口和日志验证上下文。
- [真实数据库凭据泄露到文档或版本库] → 文档仅使用配置键和占位符；真实凭据只放入本机忽略配置或环境变量。

## Migration Plan

1. 在主机的本地调试配置或环境变量中填写完整 PostgreSQL 配置、Web 监听地址和公开访问地址，不提交真实凭据。
2. 执行既有 PostgreSQL DDL，确认 `cc_admin_*` 及已迁移业务表存在，使用受限数据库账号验证读写权限。
3. 发布代码并重启主机 `DebugApplication`，确认仅该实例对局域网暴露 Web 服务。
4. 使用普通用户和两个管理员在不同浏览器/电脑访问同一主机地址，验证用户隔离、管理员一致列表和跨管理员写入可见性。
5. 如需回滚，停止共享节点并恢复先前代码；不回滚或删除 PostgreSQL 数据。开发期内存数据不纳入迁移或恢复范围。

## Open Questions

- 共享节点的生产化部署是否需要由 Cosmic 正式应用服务器替代 `DebugApplication`，以及对应的公开地址与证书策略。
- 多人独立开发时，是否为每位开发者预分配独立 PostgreSQL schema，以避免测试数据彼此干扰。
