## Context

职业测评当前通过 `InMemoryAssessmentCatalog` 提供五套内置量表。管理后台可以直接修改该对象，但服务重启后数据丢失；`/assessment/questions` 返回完整量表，提交时也缺少能证明“本次实际抽到了哪些题”的服务端凭证。题库扩容后，直接随机截取会破坏不同维度的题数平衡，并允许客户端替换题目集合。

## Goals / Non-Goals

**Goals:**
- 管理员维护的量表、题目、选项和作答题数跨服务重启保留。
- 开始测评时按维度均衡抽题，并形成用户专属、短期有效的固定作答批次。
- 提交评分只使用作答批次中的服务端题目快照。
- 重新测评时优先避免上一批题目，题库不足时可回退到必要重复。

**Non-Goals:**
- 本次不实现心理测量学常模、题目难度自适应、题目曝光率运营看板或跨版本结果等值换算。
- 本次不允许用户在答题中途手动换题。
- 本次不迁移 Spring Boot、JPA 或 Flyway 实现。

## Decisions

### 1. 使用可替换的持久化题库边界

扩展 `AssessmentCatalog`，增加量表配置保存、作答批次创建和按批次加载能力。PostgreSQL 适配使用 JSONB 保存完整量表与题目结构，并以现有内置题库作为空库初始化种子；非 PostgreSQL 环境保留进程内实现用于测试和预览。

### 2. 作答批次固定题目快照

开始测评时后端生成不可预测的 `attemptId`，保存 `userId`、`scaleId`、抽中题目快照、创建时间和提交状态。后续加载和提交必须同时匹配用户与批次；提交成功后批次标记完成，禁止重复提交。

### 3. 按主维度分层均衡抽取

以题目的 `dimensionCode` 作为分层键。对于 `EI`、`R/I` 等成对编码，保留完整编码作为题目维度组。算法先将目标题数平均分配到所有可用维度组，再把余数分配给仍有容量的组；组内使用随机顺序抽取。若某组题目不足，剩余名额在其他组中继续均衡补齐。

### 4. 降低重复而不是保证绝对不重复

创建新批次时读取该用户同量表最近一次批次的题目编号。每个维度组先随机抽取未在上次出现的题目，再从重复候选中补足。题库规模不足时允许重复，以保证配置的作答题数可满足。

### 5. 区分题库总数和作答题数

量表 DTO 新增 `poolQuestionCount`、`answerQuestionCount` 和 `attemptId`。列表页显示实际作答题数；管理后台同时显示题库总数，并允许在 1 到题库总数之间调整作答题数。删除题目导致配置超限时，后端自动将作答题数收缩到当前题库总数。

## Risks / Trade-offs

- [随机结果导致测试不稳定] -> 抽题器允许注入 `Random`，单元测试使用固定种子。
- [旧前端没有 attemptId] -> 保留服务端完整量表提交兼容入口仅供已有测试；正式 WebApp 始终走批次提交。
- [题库初始化覆盖管理员数据] -> 仅在持久化题库为空时写入内置种子，后续启动不覆盖。
- [维度编码不规范导致失衡] -> 保存题目时要求非空维度编码；历史内置题目均已有维度编码。

## Migration Plan

1. 幂等创建题库和作答批次表。
2. 空库时写入五套内置量表；已有数据不覆盖。
3. 发布后管理后台读取持久化题库，并允许设置作答题数。
4. WebApp 改为开始测评时创建批次，提交时携带 `attemptId`。
5. 验证重启持久化、维度均衡、批次不可变、跨用户隔离和重新抽题降低重复率。

## Open Questions

- 暂无。本次采用每个量表最近一次作答批次作为去重参考，后续可扩展为最近多次曝光率控制。
