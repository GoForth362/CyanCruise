## Context

当前升学中心已有方向选择、路线摘要、阶段任务、每日行动以及 `AgentPlatformStudyPlanGenerator`，也已有通用文件上传和文本提取服务。但考研智能体输入只包含基础画像和目标院校，升学中心没有资料管理入口；现有通用深造材料 PostgreSQL 适配器还是内存占位实现，不能满足本次资料持久化要求。

本次变更跨越公共 DTO、文件服务、升学中心存储、智能体调用、WebAPI 和页面。实现必须兼容 JDK 1.8 与 Cosmic，不增加 Spring Boot、JPA 或新的文件解析依赖。

## Goals / Non-Goals

**Goals:**

- 真实调用已发布的考研规划智能体/任务流，并严格校验结构化路线结果。
- 使用服务器读取的用户画像、方向、目标院校、当前日期、现有进度和用户资料构造规划输入。
- 允许当前用户在升学中心上传、查看和删除考研规划资料。
- 将资料元数据、解析正文和状态持久化到 PostgreSQL，并保持用户隔离。
- 将有效路线和每日行动持久化；失败时保留旧路线和进度。
- 保留已开始/已完成阶段，只替换未开始阶段。

**Non-Goals:**

- 不把用户资料写入公共长期知识库。
- 不在本次变更中实现扫描图片的文字识别；无可提取文字时给出清晰提示。
- 不修改就业路线生成和就业资料模型。
- 不新增浏览器直连智能体或在前端保存智能体凭据的方案。

## Decisions

### 1. 在升学中心存储边界增加规划资料，而不是复用尚未完成的通用深造 PostgreSQL 占位适配器

新增 `StudyPlanningMaterialDto` 及 `StudyCenterStorage` 的资料方法。PostgreSQL 使用 `cc_study_center_material`，与已有选择、路线和每日任务表位于同一存储边界。

原因是本次资料直接参与升学路线生成，需要与路线使用同一用户隔离和可用性边界；复用当前继承内存实现的 `PostgresqlFurtherStudyCompanionStorage` 会造成生产环境重启后丢失。未来可在通用深造存储完成后做数据归并，但本次不依赖它。

### 2. 文件二进制继续由现有 Cosmic 文件服务保存，数据库只保存文件引用和受限正文

上传流程为：文件服务保存 → 文本提取 → 升学中心保存资料记录。资料表保存文件键、原文件名、类型、大小、解析状态、解析消息、正文、方向和时间。

这样可避免在业务表重复存储二进制，并复用已有预览/下载/删除能力。正文设置单文件和总输入长度上限，防止智能体请求失控。

### 3. 智能体输入由后端聚合，不接受前端提交的画像快照

`StudyPlanApplicationService` 从画像服务、升学选择存储、现有路线和资料存储读取数据，再传给 `StudyPlanAiGenerator`。只有状态为可用且正文非空、方向为 `POSTGRADUATE` 的当前用户资料会进入输入。

智能体任务流配置使用 `cc001.agent.platform.study.postgraduate.*` 系统属性/环境配置。代码不硬编码任务流编码、智能体编码或令牌。

### 4. 显式生成必须成功通过智能体，不使用规则结果冒充

`/cc001/study-center/plan/generate` 在智能体未配置、调用失败或结构不完整时返回中文错误并保留旧路线。`ensure` 仍用于读取/准备已有页面状态，但页面“生成升学规划”走智能体接口。

### 5. 每日行动由持久化路线的当前有效阶段逐日产生

复用现有 `CareerDailyPlanApplicationService` 的按日任务持久化与完成状态。新路线成功后，已开始/已完成阶段以及对应周计划、每日进度按既有保护规则保留；未开始阶段可更新。

### 6. 资料删除同时清理业务记录，文件删除采用尽力而为

删除接口先校验资料属于当前用户，再删除文件对象；无论文件已不存在与否，均删除资料记录。若文件服务暂不可用则返回失败，不留下“页面已删除但文件仍未知”的误导状态。

## Risks / Trade-offs

- [资料正文过长导致请求过大] → 限制单文件正文和本次规划资料总长度，并向智能体提供被截断标记。
- [PDF 为扫描件无法提取文字] → 保存 `TEXT_EMPTY` 状态，页面提示用户换用可复制文字的 PDF/TXT/DOCX；该资料不进入智能体输入。
- [智能体返回 Markdown 包裹或外层 data] → 复用 JSON 提取兼容逻辑，同时对 phases、weeklyPlan、dailySuggestions 做必填校验。
- [上传成功但业务记录保存失败] → 后端组合服务在记录失败时尽力删除已上传文件，避免孤儿文件。
- [路线生成时资料被并发删除] → 生成调用使用开始时读取的不可变资料摘要；结果仍记录生成时间，下一次生成使用最新资料。

## Migration Plan

1. 部署新增 DTO、存储方法和 `cc_study_center_material` 建表语句。
2. 配置考研任务流运行参数并确认任务流已启用。
3. 部署资料 WebAPI、智能体输入聚合和页面。
4. 使用两个不同用户验证资料隔离、重启后持久化和智能体输入差异。
5. 若需回滚，前端隐藏资料入口并停止调用新增接口；保留新增表和既有路线数据，不执行破坏性删表。

## Open Questions

- 当前文件正文提取能力对 DOCX 的生产支持取决于 Cosmic 文件解析适配器；本次以前端接受类型和后端实际解析结果为准。

### 7. 将一年覆盖作为考研路线的持久化前置条件

考研任务流可能把“第一阶段先核验信息”误当作完整输出，只返回“现在至第 1 个月”。后端不能依赖页面文案推断路线完整性，因此在 `AgentPlatformStudyPlanGenerator` 中增加机器可读的十二个月、至少三个连续阶段约束，并在首次结果不完整时自动进行一次纠正调用。

结构校验要求考研路线至少包含三个阶段，且阶段时间范围中明确覆盖第十二个月或一年。历史上已经落库的单阶段路线属于不完整路线：即使该阶段已经开始，也不得作为真实路线继续保留；重新生成时先清理该路线，再整体采用通过校验的新结果。只有已验证的完整智能体路线继续遵循“已开始和已完成阶段不可替换”的保护规则。

### 8. 本地默认通过考研智能体进入任务流

考研规划与就业规划保持相同的运行方式：应用只配置智能体编码，调用 `runAgent`；考研智能体根据角色与任务设定，再调用其已绑定的考研规划任务流。`taskFlowCode` 仅作为故障排查或明确要求绕过对话智能体时的可选覆盖，不写入默认本地配置。

### 9. 升学路线只接受真实智能体结果

删除 `StudyPlanSummaryService` 中的规则兜底路线。`/cc001/study-center/plan/ensure` 仅兼容读取已有真实路线，不再创建或保存阶段、周计划和每日建议。读取到历史 `planningMode=RULE_FALLBACK`、`agentStatus=FALLBACK_READY` 或 `modelUsed=study-rule-fallback` 的路线时，后端删除该路线及其派生的每日任务，并向页面返回未生成状态。

智能体结果必须在保存前完整提供路线目标、当前情况摘要、至少三个覆盖一年的阶段、周计划、每周重点和每日行动；服务端只补充用户、方向、目标院校、版本和时间等可信元数据，不使用规则内容补齐业务字段。前端在升学路线下不生成 file 预览阶段，也不把历史单阶段兜底路线视为可更新的真实路线。

### 10. 无标记的历史单阶段考研路线同样按无效数据清理

早期规则兜底路线可能在持久化或迁移过程中缺失 `RULE_FALLBACK` 等来源标记，仅凭来源字段无法完整识别。读取或重新生成考研路线时，服务端同时校验路线是否为 `AGENT`、状态是否为 `AGENT_GENERATED`、是否至少包含三个阶段；不满足条件的历史考研路线及其每日任务直接删除，不再保护其中标为进行中或已完成的阶段。

考研重新生成后的阶段保护只在服务端对已验证的真实智能体路线执行。浏览器收到服务端结果后必须整体替换本地升学路线状态，不再二次合并旧阶段，避免本地缓存把已删除的单阶段路线恢复到页面。

### 11. 准备引导不参与真实路线按钮状态判断

升学中心的“明确升学方向、评估基础与差距、制定计划、推进关键节点”四步仅用于说明准备顺序，不代表用户已经生成规划。考研按钮与完整路线页只承认 `hasPlan=true`、`planningMode=AGENT`、`agentStatus=AGENT_GENERATED` 且至少包含三个阶段的服务端路线；浏览器本地进度和残留阶段不得把空状态改写成“更新未开始阶段”。

### 12. 十二个月覆盖按连续区间校验

路线完整性不得通过阶段文案中是否出现数字 `12` 判断，因为实际日期“2026年12月”并不表示路线已经覆盖“第12个月”。后端生成校验与历史清理优先解析“第1—2个月”等相对月份区间；没有相对区间时解析阶段中的实际年月，并要求阶段按顺序连续且首月到末月合计至少十二个月。服务端通过校验后以 `hasPlan` 返回最终结论，前端不再重复解析阶段文案。

### 13. 多段智能体消息按规划契约选择最终 JSON

对话智能体可能把任务流的用户情况整理结果和最终规划结果放在同一次回复中。通用 SDK 消息收集按就业规划既有行为选择最后一个完整 JSON 对象作为最终 `answer`；就业和考研生成器不再分别扫描原始消息。路线目标仍兼容 `targetRole` 与既有扩展字段 `targetSummary`。如果最终对象不满足规划契约，则交给既有结构校验返回统一中文错误并保护旧路线，不使用规则内容补齐。

### 14. 就业与考研复用同一个规划 JSON 解析器

金蝶 SDK 客户端负责从消息事件和任务流输出中整理最终 `answer`；就业规划和考研规划收到 `answer` 后统一使用共享解析器执行标准 JSON 解析、可选 `data` 对象解包和 `CareerPlanRecordDto` 映射。考研生成器不再维护领域专用的二次花括号扫描，避免同一份规划在就业路径可用、考研路径却因额外截取规则被拒绝。

考研任务流的兼容字段规范化及十二个月、至少三个阶段等约束仍在共享解析之后执行。这些属于考研业务校验，不改变与就业规划一致的基础 JSON 解析行为。

### 15. 前端直接信任服务端已验证的考研规划

服务端在保存和读取考研规划时已经校验智能体来源、生成状态、阶段数量以及连续十二个月覆盖，并通过 `hasPlan` 表达最终结论。浏览器不再重新解析 `horizon` 中文文案；否则前后端正则或文本格式差异会把已经落库的真实规划误判为空。前端仅在 `hasPlan=true`、`planningMode=AGENT`、`agentStatus=AGENT_GENERATED` 且至少包含三个阶段时展示考研路线，按钮与空状态共用这一判据。

### 16. 考研阶段不使用就业路线的年限筛选器

就业路线允许使用“1—3年”等跨度筛选，考研路线则由服务端固定校验连续十二个月，阶段文案还会包含“2026年7月”等实际年月。如果继续调用就业路线的年限解析，日历年份 `2026` 会被误认为规划跨度并过滤全部阶段。考研路线在通过真实规划校验后直接展示完整阶段数组；按钮状态、路线卡片和进度均使用同一数组。
