# 设计：CyanCruise 苍穹业务对象建模规划

## 总体思路

CyanCruise 当前先通过 PostgreSQL 和外部前端运行，苍穹业务对象用于把核心业务数据逐步纳入金蝶平台的权限、列表、表单、报表和后续 adapter 体系。

本变更只定义建模口径，不直接修改业务代码。建模顺序遵循“已建对象优先修正、职业发展主线优先、深造陪伴随后、Agent 可解释性对象最后补齐”的原则。

## 命名设计

平台对象和字段以金蝶设计器保存后的标识为准：

- 对象编码：`v620_cc_xxx`，创建弹窗中不得超过 25 个字符
- 表名：`tk_v620_cc_xxx`
- 字段标识：`v620_xxx`

文档中不再使用驼峰字段标识作为正式字段标识。对于代码侧仍使用 snake_case 的逻辑字段，通过“逻辑字段”列说明映射关系。

对象编码和表名都需要控制长度；较长对象使用稳定缩写，例如 `assessment` 使用 `assess`，`diagnosis` 使用 `diag`，`capability` 使用 `cap`，`context` 使用 `ctx`，`further_study` 使用 `study`。最终以金蝶设计器可保存的实际编码和表名为准。

例如：

| 平台字段 | 逻辑字段 |
| --- | --- |
| `v620_userid` | `user_id` |
| `v620_targetrole` | `target_role` |
| `v620_resultjson` | `result_json` |

## 已建对象修正

### 用户职业画像

“用户职业画像”是基础资料，用于保存用户长期职业画像汇总。它已经在平台建立，文档只记录已经实际采用的字段：用户标识、目标岗位、当前阶段、个性化等级、完整度分数、画像内容、准备度内容、证据内容、Agent摘要和最近Agent执行ID。

### Agent执行记录

“Agent执行记录”是单据，用于保存一次 Agent 调用的审计信息。当前阶段只需要单据头字段，不建立“执行步骤”分录。执行步骤如果未来需要追踪工具调用链，再作为单独 change 增补。

## 待建对象分组

### 职业发展

职业发展对象包括职业测评记录、简历档案、简历诊断、今日任务、职业计划和模拟面试。这些对象承接现有 DTO、WebAPI 和 PostgreSQL 存储语义。通用“AI助手会话”不作为当前主业务模型；CyanCruise 的 AI 能力按任务型 Agent 建模。

### 深造陪伴

深造陪伴对象统一承载考研、保研和留学，而不是为三类方向复制三套对象。通过 `v620_track` 区分方向，通过 `v620_recordtype` 区分生成结果类型。

### Agent 运行管理

Agent能力定义登记主调度、用户画像、信息检索、学业规划、就业规划、简历优化、面试辅导、考研规划、保研规划和留学规划等专项 Agent 的职责、思维链路和关键工具。Agent执行记录记录一次执行的输入、输出、状态和消耗；Agent上下文引用记录一次执行读取了哪些业务对象。上下文引用可以在主业务对象建完后再建。

## 数据安全

- 大文本 JSON 字段不得保存 access token、Authorization header、签名 URL、数据库密码或客户私有凭据。
- 文件字段只保存稳定 `file_key` 或平台文件 ID。
- 用户归属必须使用结构化 `v620_userid` 字段，不得只隐藏在 JSON 中。

## 后续接入

后续 Java adapter 接入时，应在 adapter 层显式维护平台字段和代码逻辑字段的映射，不要求 `base-common` 或 `base-helper` 依赖 Cosmic 平台类型。
