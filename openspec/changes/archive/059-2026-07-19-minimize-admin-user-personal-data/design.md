## Context

管理员用户列表当前直接返回 `AdminUserDto`，该对象同时承担内部持久化和外部展示职责。应用服务还会从职业画像补写学校、专业，导致账号治理接口暴露与账号状态管理无关的个人数据。平台组织 ID 则是服务端组织隔离所需字段，但当前单组织页面直接展示原始编号，没有用户价值。

## Goals / Non-Goals

**Goals:**

- 管理员用户列表和详情接口遵循数据最小化，只暴露账号治理字段。
- 停止按学校、专业搜索及从职业画像复制这些信息。
- 保持平台组织 ID 对组织隔离、组织看板和广播范围的内部作用。

**Non-Goals:**

- 不删除用户本人保存的学校、专业画像。
- 不改变管理员权限判断、用户禁用和组织隔离规则。
- 不进行历史个人画像数据清理迁移。

## Decisions

### 决策一：在应用服务出口创建隐私安全副本

`AdminUserDto` 仍用于内部治理存储，避免破坏现有 PostgreSQL 载荷和组织逻辑；`listUsers` 与 `userDetail` 返回前创建副本，仅保留用户 ID、昵称、状态、管理员标识和治理时间字段。相比直接删除 DTO 字段，该方式不会影响组织看板和内部存储兼容性。

### 决策二：用户管理不再读取职业画像

移除列表中的画像补写流程，关键字匹配仅使用昵称与用户 ID。个人画像继续由面向用户本人的职业画像能力管理，不进入账号治理视图。

### 决策三：页面同步删除无关列

用户管理表仅显示用户、身份类型、状态和操作；搜索提示与空状态只描述姓名和用户 ID。原始组织 ID 不再展示，但后台请求上下文仍携带组织标识。

## Risks / Trade-offs

- [管理员无法按学校或专业筛选用户] → 这是有意收紧的隐私边界；如未来确有合规统计需求，应设计单独的聚合报表和授权能力。
- [内部 DTO 仍含个人字段] → 外部接口通过安全副本清空字段，并由回归测试约束；后续可独立迁移为内部实体和公开 DTO。
- [历史治理记录仍可能保存旧字段] → 页面和接口不再暴露，避免在本次变更中进行高风险数据迁移。

## Migration Plan

1. 部署应用服务与 WebApp。
2. 验证用户列表和详情响应不包含学校、专业、组织 ID 的值。
3. 验证管理员仍可搜索姓名或用户 ID、禁用和恢复用户。
4. 如需回滚，可恢复旧应用服务映射和页面列，不涉及数据库结构回滚。
