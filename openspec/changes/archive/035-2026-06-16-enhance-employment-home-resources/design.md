## Context

CyanCruise 已有就业洞察和职业资源 WebAPI：`/cc001/career-employment/insight/get` 与 `/cc001/career-employment/resources/list`。当前就业首页只调用本地 `featureGroups` 渲染工具卡，未消费资源流，因此用户看不到就业资讯、职业指导文章或公开就业服务入口。

项目约束要求兼容 JDK 1.8，不引入破坏 Cosmic/KDDT 模板的依赖，不硬编码本地环境路径，也不能提交数据库密码。由于 Cosmic 运行时直接爬取外部网页会带来网络、合规、稳定性和依赖风险，本次采用“可追溯资源卡 + 前端聚合展示”的方式先完善体验。

## Goals / Non-Goals

**Goals:**

- 就业首页在首屏下方直接展示“就业资讯与文章”和“就业洞察摘要”。
- 资源内容来自既有资源 API，前端在接口失败时展示可理解的降级状态并保持工具入口可用。
- 默认资源种子包含公开就业服务、求职指导、招聘信息和面试/简历类文章入口，后续可替换为 PostgreSQL 或运营配置来源。
- 保持 JDK 1.8 与现有无新增依赖策略。

**Non-Goals:**

- 本次不实现运行时网页爬虫、RSS 订阅、外部站点 HTML 解析或自动定时采集。
- 本次不新增数据库表，不存储第三方网页内容全文。
- 本次不改 `/cc001/careerloop` 兼容路由、历史 localStorage key 或既有类名。

## Decisions

1. 就业首页直接消费既有资源 API，而不是创建新接口。

   理由：后端已经有 `CareerResourceFeedDto`，可表达文章、视频、咨询和职业路径。复用它可以减少 API 面并保持路由契约稳定。

2. 资源默认来源采用人工配置的公共链接，而不是运行时联网抓取。

   理由：CyanCruise 当前部署在 Kingdee Cosmic 二开工程中，直接抓取外部网页会引入网络可用性、HTML 结构变化和依赖兼容风险。人工配置链接能先满足“用户可看可用”的体验，并为后续定时采集留出 `CareerResourceStorage` 替换点。

3. 前端把就业首页作为组合工作台渲染。

   理由：用户进入“就业”后应同时看到工具、资讯和洞察，而不是再进入隐藏调试页。资源区和洞察区均为页面内容，不做嵌套卡片，保持现有 CyanCruise 风格。

## Risks / Trade-offs

- 外部链接可能过期或内容变动 → 默认种子只放公开就业服务和职业指导入口；后续可通过 PostgreSQL/管理后台配置替换。
- 首次进入就业页需要额外请求资源和洞察 → 前端分离资源、洞察状态，失败时只降级对应区域，不阻断工具入口。
- 资源内容不是实时抓取 → 本次明确为配置型资源流；实时采集另开 change 处理调度、审核、缓存和合规。

## Migration Plan

1. 增强就业首页渲染逻辑和状态字段。
2. 扩充默认资源种子为 CyanCruise 命名的公开就业资讯/服务入口。
3. 更新前端资源版本号。
4. 使用 `node --check`、路由校验、OpenSpec strict 校验和 JDK 8 Gradle 测试验证。
