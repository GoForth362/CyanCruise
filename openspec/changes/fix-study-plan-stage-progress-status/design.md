## Context

升学路线的阶段状态目前有三类来源：智能体返回的 `phase.status`、浏览器 `localStorage` 保存的阶段勾选进度、服务端每日任务完成记录。页面还会把第一条未完成阶段当作“当前阶段”，即使完成数为零也显示“进行中”；服务端则直接保护智能体返回为 `IN_PROGRESS` 的阶段。这两处推断都缺少用户真实执行证据。

## Goals / Non-Goals

**Goals:**

- 以至少一项真实勾选或持久化完成记录作为“进行中”的最低条件。
- 零进度阶段在页面显示为“未开始”，并允许重新生成。
- 重新生成时保留有真实进度及已完成的阶段。
- 对考研、保研、留学三类升学规划使用同一规则。

**Non-Goals:**

- 不改变路线图阶段、动作和达成项的数据结构。
- 不重做现有浏览器进度存储。
- 不调整就业路线的阶段保护策略。

## Decisions

1. 页面阶段状态只依据任务计数：完成数为零时显示“未开始”；完成数大于零且小于总数时显示“进行中”；全部完成时显示“已完成”。`currentPhaseId` 和智能体提供的 `IN_PROGRESS` 只用于计划顺序，不再单独驱动状态标签。
2. 升学规划刷新状态读取当前路线对应的 `progressState`。只有本地已勾选进度、服务端 `COMPLETED` 状态或持久化每日任务完成证据才保护阶段。
3. 生成请求增加 `startedPhaseIds`，由页面根据阶段勾选结果计算。服务端将它与持久化每日任务完成记录合并后判断受保护阶段，避免只依赖浏览器，也避免把智能体预填状态当作真实进度。
4. 新生成且未被保护合并的阶段统一规范为 `NOT_STARTED`。智能体负责生成计划内容，不负责声明用户已经执行了计划。
5. 每日任务状态重算时，阶段完成数为零必须写入 `NOT_STARTED`，至少完成一项才写入 `IN_PROGRESS`，全部完成才写入 `COMPLETED`。

## Risks / Trade-offs

- [旧计划仅有 `IN_PROGRESS` 但没有任务完成证据] → 按未开始处理，使此前被错误锁定的第一阶段重新可刷新。
- [浏览器本地勾选未跨设备同步] → 本次生成请求显式传递当前浏览器进度；每日任务完成记录继续提供跨设备持久化证据。
- [客户端伪造阶段 ID] → 服务端只接受当前已保存路线中存在的阶段 ID，忽略未知值。
- [前后端版本短暂不一致] → 新请求字段为可选；旧客户端仍可调用，新服务端使用持久化证据兜底。
