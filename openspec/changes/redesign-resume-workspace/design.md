## Context

`resume` 页面当前由 `renderResumePage` 组合 `resumeFormPanel` 与 `resumeListPanel`，事件统一由 `bindResumeEvents` 绑定。创建表单同时承载元数据、PDF 上传和正文粘贴，记录卡则通过 `resumeItem` 提供预览、诊断和删除。现有业务闭环完整，但缺少资产总览、过程引导与清晰的记录状态表达，在窄屏和长正文场景下阅读压力较大。

## Goals / Non-Goals

**Goals:**

- 使用真实简历列表建立资产总览，并突出最有价值的下一步。
- 将创建流程拆分为用户可理解的基础信息、PDF、正文和保存步骤。
- 让每份记录的岗位、文件、诊断和更新时间可以快速扫描。
- 保留全部 DOM 标识、数据属性、提交状态和局部错误恢复逻辑。
- 消除窄屏横向滚动，支持长标题、长岗位、长时间文本和键盘操作。

**Non-Goals:**

- 不增加在线富文本简历编辑器或模板系统。
- 不改变简历、文件、诊断或画像接口。
- 不修改上传大小限制、正文提取规则、数据库或后端服务。
- 不重构简历诊断页面。

## Decisions

### 使用独立简历页面模式

仅在 `resume` 路由添加 `resume-workspace-mode`，所有新样式限定在该模式下，避免影响诊断页和其他复用的表单、列表、按钮组件。

### 总览只使用真实记录

从 `state.resumes` 计算简历总数、已关联 PDF 数和已有诊断分数的记录数；无记录或列表错误时显示真实的零值或错误状态，不生成演示指标。

### 保留单表单和现有事件标识

继续使用 `resumeForm`、`resumeFileInput`、`uploadResumeFileButton`、`resumeFileKey`、`resumeParsedContent` 和现有提交函数，只增加步骤包装与说明。这比拆成多个独立表单更安全，可避免改变上传后草稿保留和最终创建请求。

### 记录卡使用状态标签而不是连续文本

`resumeItem` 继续输出相同的预览、诊断和删除数据属性，但将目标岗位、PDF、诊断分数和更新时间分成状态标签与元信息区。主要操作为“继续诊断”，预览为次要操作，删除保持危险操作。

### 窄屏使用单列信息流

桌面端保留创建区与记录区的主次布局，窄屏改为单列；按钮可换行，文本允许任意位置换行，文件输入和上传按钮不依赖固定宽度。

## Risks / Trade-offs

- [共享组件样式被覆盖] → 所有增强样式限定在 `resume-workspace-mode`。
- [重排后事件无法绑定] → 保留所有既有 ID、`data-preview-file`、`data-diagnose-resume` 和 `data-delete-resume`。
- [空列表时总览误导] → 指标只从真实 `state.resumes` 计算，空列表明确显示零值和创建入口。
- [长文件状态撑开卡片] → 对网格子项设置 `min-width: 0`，并允许文本换行。

## Migration Plan

1. 增加页面状态类和真实简历总览。
2. 重构创建表单和记录卡语义结构。
3. 增加专属青色样式、窄屏和焦点适配。
4. 更新静态契约与缓存版本，完成视觉和构建验证。

回滚仅需恢复前端渲染、样式与资源版本，不涉及数据迁移。
