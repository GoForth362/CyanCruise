## Context

现有升学中心已经具备方向选择、考研智能规划、资料上传、完整路线和今日行动。`AgentPlatformStudyPlanGenerator` 也预留了 `cc001.agent.platform.study.recommendation.*` 配置，但当前规划表以 `user_id` 为唯一键、每日任务主键为 `user_id + task_id`，存储接口也只按用户读写，导致同一用户切换考研和保研时无法可靠保留两套数据。前端资料区只在考研方向展示，真实规划判据和空状态文案也只针对考研。

本次变更跨越公共契约、PostgreSQL 迁移、应用服务、智能体适配、WebAPI 和静态页面，必须兼容 JDK 1.8、Cosmic 运行时和现有考研数据。

## Goals / Non-Goals

**Goals:**

- 同一用户的考研、保研规划、每日任务和规划资料按具体方向隔离并持久化。
- 保研复用考研的真实智能体规划链路、进度保护、JSON 解析和每日任务拆解。
- 未配置智能体或返回失败时只给出可恢复错误，不生成任何兜底路线、示例阶段或虚假今日行动。
- 前端在保研方向展示独立资料、真实规划、完整路径和今日行动。
- 预留独立保研智能体配置，配置编码后无需修改业务代码即可启用。

**Non-Goals:**

- 不在本次实现留学智能规划。
- 不创建保研规则模板或静态推荐路线。
- 不把用户上传资料写入公共知识库。
- 不硬编码智能体编码、任务流编码或令牌。

## Decisions

### 1. 存储键扩展为用户与具体升学方向

`StudyCenterStorage` 的规划与每日任务方法增加 `direction` 参数；内存实现使用复合键。PostgreSQL 规划表主键升级为 `(user_id, direction)`，每日任务主键升级为 `(user_id, direction, task_id)`，所有查询、更新和删除同时约束用户与方向。

相比为保研再建一套平行表，复合键能让三种升学方向复用同一契约与索引，同时保持数据隔离。资料表已经包含 `direction`，补充按方向校验删除和读取归属即可。

### 2. 采用兼容迁移而不是破坏性重建

初始化时检测并迁移旧表主键：历史规划根据记录中的 `direction` 保留为对应方向，空方向按已有考研历史语义归入 `POSTGRADUATE`；每日任务同样补齐方向。迁移通过临时表复制后替换，避免生产数据丢失。独立 SQL 脚本与运行时初始化保持一致。

### 3. 保研复用通用生成器但使用独立配置与语义

继续由 `AgentPlatformStudyPlanGenerator` 按方向选择配置。保研使用独立的 `cc001.agent.platform.study.recommendation.*` 配置，并与考研保持一致：配置 `taskFlowCode` 时优先直连任务流，未配置时才使用 `agentNumber` 调用智能体。输入包含 `direction=RECOMMENDATION`、用户画像、目标院校、既有进度和仅属于保研方向的可用资料。任务流编码只通过运行环境配置注入，不在业务代码中硬编码。

输出继续使用共享规划 JSON 契约。保研要求至少包含非空阶段、本周计划、每日建议和保研语义目标；不套用考研初试十二个月专属校验。智能体未配置、超时或格式错误时抛出业务错误并保留旧规划。

### 4. `ensure` 只读，不生成兜底数据

保研与考研一致：`ensurePlan` 只返回已经持久化且来源可信的规划；无规划时返回 `hasPlan=false`。只有显式生成接口可以调用智能体。今日行动只从真实规划持久化派生，无真实规划时返回空任务和明确提示。

### 5. 页面状态由当前方向驱动

升学中心资料列表、生成按钮、完整规划和今日行动均读取当前保存的方向。保研方向显示“生成保研规划”“尚未生成真实保研规划”等普通中文；切回考研时恢复考研规划和任务。浏览器不合并不同方向缓存，也不构造本地预览阶段。

## Risks / Trade-offs

- [旧 PostgreSQL 主键迁移失败] → 在事务中创建新表、复制并校验后再替换；保留幂等初始化和独立 SQL 验证。
- [任务 ID 在不同方向重复] → 主键和所有更新条件加入 `direction`，避免跨方向误更新。
- [保研智能体尚未配置] → 生成接口返回清晰配置错误，页面保留空状态和已存在数据。
- [保研输出混入考研语义] → 输入固定方向并增加方向一致性与基础结构校验，不用规则内容修正结果。
- [前端缓存恢复另一方向阶段] → 服务端响应整体替换当前方向状态，前端进度键加入具体方向。

## Migration Plan

1. 先部署兼容存储契约和 PostgreSQL 复合键迁移，验证历史考研规划可读。
2. 部署方向化应用服务、智能体预留配置和 WebAPI。
3. 部署保研升学中心、完整规划及今日行动页面资源。
4. 在未配置保研智能体时验证空状态与失败不落库，再配置智能体编码完成真实生成验证。
5. 回滚时保留复合键表和数据；旧代码不应在迁移后重新部署，以免只按用户读取造成语义不明确。
